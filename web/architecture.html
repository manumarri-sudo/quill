Quill The security gate for AI coding agents

Defense in depth, in two layers

Catch it at the gate.
Contain it at the floor.

Quill sits between an AI coding agent and your machine, so before any command runs or any file is written a deterministic gate classifies the action and answers allow, ask, or deny, and because there is no language model anywhere in that decision, the gate cannot be talked out of its answer. When something does slip past the gate, a kernel floor built into the operating system contains the blast radius, and every decision either way is written to a tamper-evident ledger.

Two layers, two different jobs

The gate is smart and the floor is dumb, and that division is the whole point, because a clever classifier catches most manipulation while a simple kernel rule contains whatever the classifier misses, and the two together cover far more than either could alone.

Layer 1 · Application

The Smart Gate

Catches manipulation before it reaches the operating system. It runs as a hook inside the agent and as a proxy in front of the agent's tools, and it answers in well under a tenth of a second.

Deterministic risk classifier

Every action is scored LOW, MEDIUM, HIGH, or CRITICAL by compiled patterns, never by a language model, so the gate cannot be prompt-injected or jailbroken into changing its own answer.

classify_command() · LOW/MED then allow, HIGH then ask, CRITICAL then deny covers rm -rf, DROP TABLE, git push --force, and the documented evasions: base64-to-shell, heredocs, $(which rm), variable-assembled $a$b
Secret scanner on file writes

A write that embeds an AWS, OpenAI, Anthropic, GitHub, or Stripe credential is escalated to CRITICAL and denied before the key ever lands in source.

scan_args() on Edit/Write/NotebookEdit then CRITICAL, overrides any policy downgrade
Gate self-tamper guard

An edit aimed at Quill's own config or the host agent's hook settings is treated as an attempt to disable the gate from the inside, so it is denied as CRITICAL.

_is_gate_config_path() then ~/.claude/settings.json, ~/.cursor/hooks.json, ~/.quill/config.toml, ~/.quill/key
Lethal-trifecta taint tracker

It watches for the dangerous combination of untrusted input, access to private data, and a path out of the machine, and when one action would close all three in a session it stops to ask you first.

would_close_trifecta() then untrusted input + private data + exfil vector then escalate to CRITICAL deny + approve token
MCP tool-poisoning detection

Tool descriptions are checked for invisible Unicode, injection-shaped phrasing, and schema that drifts after first trust, which is how a poisoned tool tries to rewrite the agent's instructions.

MCP proxy inspects tool descriptions for hidden Unicode, prompt-injection markers, and post-trust schema drift (rug-pull)
Layer 2 · Kernel

The Floor

Contains the damage when manipulation slips past the gate. It does not try to be clever about what an action means, it simply enforces hard limits on the real syscall, which is exactly why a regex evasion or a write-then-run trick cannot route around it.

Seatbelt sandbox

It limits where the agent can write and closes off every path that data could use to leave the machine, and because that confinement is inherited, any program the agent launches is boxed in by the same walls rather than slipping past them.

sandbox-exec profile then write paths allow-listed, outbound network denied inheritance closes the write-then-run loophole: python3 deploy.py runs inside the same sealed sandbox
Endpoint Security extension

It hooks the operating system's own checkpoint for running programs and touching files, so an exec or a file syscall that should not happen is refused by the OS itself, beneath anything the interpreter could do.

macOS Endpoint Security (ES) client authorizes ES_EVENT_TYPE_AUTH_EXEC and file-open events the verdict happens on the real syscall, not on what the command looks like
Why this is the load-bearing layer

A pattern list can always be evaded by an input it has not seen, but the kernel does not read patterns, it reads syscalls, so the floor holds even against an action whose surface form is brand new.

What we will and will not claim

The defensible claim is narrow and we keep it that way: no known bypass for a non-root agent without an OS zero-day.

That is a different and smaller statement than saying the gate cannot be bypassed at all, and the difference matters, because a security tool that oversells its walls leads an operator to take risks they would never take if they saw the walls clearly. The gate has no language model in its decision path, so it cannot be jailbroken, and that is true, yet it is not the same as saying it cannot be bypassed, which is why the kernel floor exists in the first place.

True at the same time

Quill is a strong defense-in-depth layer and an excellent, tamper-evident record of everything the agent attempted, allowed, and refused.

Also true at the same time

The application layer alone is not a hard boundary against a determined adversary, which is the gap the kernel floor is built to close.
