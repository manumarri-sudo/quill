# Quill Audit — Full Issue → Fix Plan

Total findings: 58 — Critical:2, High:14, Medium:20, Low:22

## 1. [Critical] (security) `quill off` self-disable hole: agent can neuter the entire gate (same bug class c9b522a just fixed for approve) **[LAUNCH BLOCKER]**
- **Issue:** c9b522a fixed `quill approve` so that an UNAVAILABLE Touch ID sensor REFUSES by default, with the explicit rationale (commit message + cli.py:3943 comment) that 'an agent's own process has no GUI/Secure-Enclave session, so touchid.is_available() is always False there' — letting it self-approve. The symmetric path `quill off` (off_cmd, cli.py:1368) calls `_require_disable_auth` (cli.py:1340), which at lines 1355-1361 does the OPPOSITE: when `touchid.is_available()` is False it prints a yellow warning and RETURNS (falls open). tests/test_disable_auth.py:48 even codifies this as `test_no_biometry_falls_open_with_warning`. So a hijacked/injected agent runs `quill off --for 24h` in its own (sensor-less) context, the guard does not fire, and the gate is FULLY off — including the CRITICAL/never-downgradable class (rm -rf, DROP TABLE, force-push) — for up to 24h. Every subsequent destructive call passes with a `gate_paused` marker. This defeats the headline 'even you cannot walk past it until I approve' model exactly as the approve hole did. The pause check also runs BEFORE self_test in main() (claude_code.py:1405-1415), so it's a fully agent-reachable kill switch.
- **Fix:** Apply the c9b522a pattern to `_require_disable_auth`: on `touchid.is_available() == False`, REFUSE by default (raise typer.Exit) rather than falling open. Add an explicit, logged `--no-biometric` opt-in flag for genuine headless/SSH operators (mirroring `quill approve`), so the agent's default path is closed while the human SSH operator can still deliberately opt in. Update test_no_biometry_falls_open_with_warning to assert refusal-by-default + opt-in, and add a regression test mirroring the approve self-disable test.

## 2. [Critical] (dx) Hero install command `uvx quillx start` is broken against the published PyPI wheel **[LAUNCH BLOCKER]**
- **Issue:** README.md lines 17-21, 142-143 lead with `uvx quillx start` as THE one-command install. I downloaded the published artifact (`quillx-0.2.0a5-py3-none-any.whl`) and inspected its `entry_points.txt`: it declares ONLY `quill = quill.cli:app`, NOT `quillx`. `uvx quillx start` instructs uv to run an executable literally named `quillx`, which does not exist in the published wheel, so it fails with a 'failed to find executable' error. The fix is present in the source `pyproject.toml:92` (`quillx = "quill.cli:app"`) but has NOT been published — the source tree's editable install does expose both `quill` and `quillx`, masking the bug locally. The README also never documents the `uvx --from quillx quill start` fallback. The single most prominent line in the entire project fails for every new PyPI user on a fresh machine.
- **Fix:** Cut and publish a new release (a6) that includes the `quillx` console script before going public — the project must not launch with its headline command pointing at an unpublished fix. Verify post-publish by running `uvx quillx start` in a clean environment / CI smoke test. Until that release exists, do not put `uvx quillx start` in the README; either ship the release first or change the hero command to the verified-working `uvx --from quillx quill start`.

## 3. [High] (security) `quill approve --no-biometric` remains an agent-reachable self-approval path after c9b522a **[LAUNCH BLOCKER]**
- **Issue:** c9b522a closed the IMPLICIT fall-through (unavailable sensor now refuses by default). But the EXPLICIT opt-out `--no-biometric` flag (cli.py:3846) is still a normal CLI flag any process can pass, and `quill approvals list` (cli.py:4023-4046) prints the FULL raw `ap.token` (line 4040), not a hashed id. So an agent can read its own pending token from `quill approvals list` and run `quill approve <token> --no-biometric`, which sets skip_biometric=True (cli.py:3915) and approves with zero human involvement — the same self-approval the fix was meant to prevent, just one flag away. The flag is intended for headless operators, but nothing distinguishes a headless human from a headless agent. This is the residual edge of the same hole.
- **Fix:** Make `--no-biometric` itself require an out-of-band proof that the caller is the operator, OR gate it behind an env/config that an agent in-session cannot plausibly set, OR (simplest) have `quill approvals list` print only a HASHED token id and require the operator to confirm via `--latest` + Touch ID (the path c9b522a added), removing the raw-token-copy flow entirely for the biometric-available case. At minimum, log `--no-biometric` approvals at a distinct, loud audit level and document that this flag re-opens the self-approval surface on headless boxes.

## 4. [High] (architecture) Audit chain detects edits/insertions but NOT tail-truncation (no length or head commitment)
- **Issue:** verify_chain (src/quill/audit.py:251-276) walks prev_mac links forward from prev_mac_hex="" and reports per-line failures. Because each entry only commits to the PRIOR entry's mac, deleting the last N lines of ~/.quill/audit.log.jsonl yields a chain that still verifies cleanly with zero failures — the verifier has no committed total count and no signed head/anchor to compare against. An attacker (or a crashed write) that removes the trailing evidence of an attack passes `quill audit verify`. The module docstring (lines 12-13) claims 'any modification or insertion breaks the chain at the next verify' — true for edits and insertions, false for truncation. Separately, the genesis entry's prev_mac is the empty string with no installation-bound anchor, so on a fully-compromised host the whole log can be re-forged from scratch under a rotated key (acknowledged as limit 5 in SECURITY-MODEL.md, but truncation is NOT called out there).
- **Fix:** Persist a signed head pointer (last mac + monotonically increasing entry count) to a separate file updated under the same flock, and have verify_chain take an expected_count/expected_head and flag a shortfall as a failure. Document truncation explicitly alongside limit 5 in SECURITY-MODEL.md. For real anchoring, periodically emit the head mac to an append-only external sink (the 'externally anchored' option the doc already gestures at).

## 16. [High] (testing) No test for audit-log trailing truncation — a valid shorter chain passes `verify` silently
- **Issue:** verify_chain (src/quill/audit.py:251-276) walks from line 1 and checks each entry links to the prior MAC. tests/test_audit.py covers modify (line 51) and insert (71) tampering, reopen continuity, the >4KB tail edge cases, mode 0o600, and concurrent writers — strong. But nothing tests deleting the last N entries (or all of them). Because the chain has no stored length/high-water-mark and no sealed tail anchor, a truncated log is a perfectly valid shorter chain: verify returns zero failures. This needs neither machine nor key compromise — just `head -c` on the file an agent can already write. SECURITY-MODEL.md (119-121) honestly says tamper-evident-not-proof and that a compromised host can rewrite log+key, but truncation is a weaker, un-flagged, untested attack, and README line 42 leans on 'append-only storage' / 'cryptographic hashing' as evidence-grade.
- **Fix:** Add a test that truncates trailing lines and documents the resulting behavior (verify passes — that's the limit). Then either (a) record an expected event count / periodic sealed checkpoint so truncation is detectable, or (b) add an explicit 'truncation is undetected locally' line to SECURITY-MODEL.md and stop implying append-only storage closes that gap in README.

## 17. [Medium] (security) Audit log stores short secrets and credential-bearing command lines verbatim at rest
- **Issue:** `_redacted_input` (claude_code.py:416-428) ONLY truncates string args longer than 200 chars; it performs NO secret redaction. A secret under 200 chars (most API keys, passwords, bearer tokens, DB DSNs) inside a Bash command or Write payload is written verbatim into ~/.quill/audit.log.jsonl as `args_preview`. Separately, `_summarize_call` (claude_code.py:304-322) captures the raw command line into the `what` field — so `mysql -u root -phunter2`, `curl -H 'Authorization: Bearer sk-...'`, `aws ... --secret-access-key ...` land in the log in full. The secret SCANNER (scan_args) detects+blocks secrets in Write payloads, but does not redact what gets logged, and does not run on Bash command strings for logging purposes. The log is 0o600 (good, so this is local-at-rest, mitigated by 'attacker-with-shell-already-wins' per the threat model), but the log is also the artifact users are told to hand to auditors/insurers, raising the blast radius.
- **Fix:** Add a redaction pass over both `args_preview` and `what` before writing to the audit log: run the existing secrets scanner's detectors and replace matched spans with `[REDACTED:<type>]`. Also redact obvious inline-credential CLI shapes (`-p<pw>`, `--password`, `Authorization: Bearer`, `aws_secret_access_key`, connection-string passwords). Keep enough to preserve the evidentiary value (tool, verb, target) without storing the secret material.

## 20. [Medium] (architecture) run_hook is a ~470-line sequential mutation of a single decision object with no shared policy core
- **Issue:** src/quill/adapters/claude_code.py:run_hook (lines 755-1228) classifies once via decide(), then re-binds `decision` through six independent best-effort blocks in sequence: trust-scope downshift, promoted-override downshift, bypass-mode downshift, session-memory recall, one-shot approval consume, then trifecta RE-escalation. Each block is its own `with contextlib.suppress(Exception)` and re-derives pattern_id / re-loads stores via function-local imports. The ordering is load-bearing (e.g. trifecta must run AFTER the allow-downshifts to re-deny, session-approval must not fire on critical) but that ordering lives only in comments, not in types or tests-of-record, and several blocks re-implement the same `'default risk for' in decision.reason` string sniff to decide eligibility — a fragile coupling to a human-readable reason string. A future edit to decide()'s reason text silently disables every downshift guard.
- **Fix:** Extract a single ordered pipeline of named DecisionModifier steps (each pure: (decision, context) -> decision) with the eligibility predicate keyed off a structured field on HookDecision (e.g. decision.source == 'default_builtin'), not a substring of decision.reason. This makes the load-bearing order explicit and unit-testable per step.

## 21. [Medium] (architecture) Eligibility for downshifts keys off the human-readable reason string ('default risk for')
- **Issue:** Three separate guards in run_hook (lines 819-824, 864, 893-897) gate on `"default risk for" in decision.reason` to decide whether a downshift may apply. That string is produced in classify_event at line 255 (`reason = f"default risk for {tool_name}"`).
